FIXED R5-01: Username Enumeration via WebAuthn Login/Begin was M-H

What changed: BeginLogin now calls BeginDiscoverableLogin() for all cases — known user, unknown user, empty username, no body. The response is always HTTP 200 with an identical discoverable-credential challenge. No allowCredentials field is ever returned.

Source: internal/service/webauthn_service.go:243-267 — both branches use BeginDiscoverableLogin().
Live: 10 requests across 4 payload types — all return identical 194-byte responses. Timing delta is sub-millisecond (100–115ms range), not practically exploitable over network.

Source review noted two residual items: (1) a timing side channel from the user DB lookup on line 255, and (2) a dead code branch for ErrWebAuthnNoCredentials at webauthn.go:96-97. Live testing confirmed the timing delta is not observable over network latency. The dead code is a cleanup item (see R6-04 below), not a vulnerability.

FIXED R5-02: TOCTOU Race on Challenge Consumption was MED

What changed: Challenge store implements Consume() using DELETE ... RETURNING — a single atomic SQL statement that deletes and returns in one operation. No transaction needed; no race window exists.

Source: internal/store/webauthn_challenge_store.go:65-89 (atomic Consume).
internal/service/webauthn_service.go:153 (FinishRegistration), :298 (FinishLogin) — both use Consume.
Live: Two concurrent requests with same challenge_id — first got HTTP 401 (reached crypto verification), second got HTTP 400 ("challenge expired or not found"). Atomic consume confirmed.

// Detailed Residual Findings

LOW R6-01: Login JS Still Sends challenge_id in Query Params

The headerOrQuery helper and registration JS were updated to use the X-Challenge-ID header, but the shared login JS was missed. webauthn.js:73 still constructs the URL as '/api/v1/webauthn/login/finish?challenge_id=' + challengeId. The OAuth prompt handler (handler/oauth/handler.go:491-492) also reads exclusively from query params.

This means login ceremony challenge IDs appear in server access logs, Cloudflare edge logs, and browser history. Registration ceremonies are clean.

Recommended Fix

Update webauthn.js to send X-Challenge-ID via header, matching the pattern already used in passkey-register.js. Update the OAuth handler to use the headerOrQuery helper.

LOW R6-02: Origin Check Allows Absent Origin Header

The admin passkey login Origin check at handler/admin/handler.go:260 uses if origin != "", meaning requests without an Origin header bypass the check entirely. While modern browsers send Origin on cross-origin POST requests, some configurations (certain proxies, non-browser clients, older browsers) may omit it.

Practical risk is low because SameSite=Strict on the admin session cookie already blocks cross-site submissions in modern browsers, and the endpoint also requires a valid JWT access token.

Recommended Fix

For POST requests, require the Origin header to be present. If absent, fall back to checking the Referer header. Only skip validation for requests with neither header if the method is GET or HEAD.

LOW R6-03: OAuth authorizePasskey Has No Origin Check

The authorizePasskey handler (handler/oauth/handler.go:216-271) accepts a POST with access_token and issues an OAuth authorization code, but performs no Origin header validation. Unlike the admin endpoint which received a fix, this endpoint was not updated.

Practical risk is low: the endpoint requires a valid JWT access token (15-min TTL, HS256-signed), and exploitation requires both a stolen token and a cross-origin attack vector. However, adding Origin validation is straightforward defense-in-depth.

Recommended Fix

Apply the same Origin validation used on the admin passkey login endpoint. Ideally extract the Origin check into shared middleware so all passkey POST endpoints are covered uniformly.

INFO R6-04: Dead Code Branch for ErrWebAuthnNoCredentials

At handler/api/webauthn.go:96-97, the loginBegin handler still has a case for service.ErrWebAuthnNoCredentials. The service no longer returns this error from BeginLogin (it was eliminated as part of the R5-01 fix). This is dead code — not exploitable, but should be removed for clarity.

Source: internal/handler/api/webauthn.go:96-97 (unreachable branch).
INFO R6-05: HSTS Max-Age Still 30 Days

The live server returns Strict-Transport-Security: max-age=2592000 (30 days). The application code sets 2 years, suggesting a Cloudflare override. Noted in Round 5 (R5-09) and still present. Does not affect core security posture but falls short of the commonly recommended 1-year minimum.

Live: Strict-Transport-Security: max-age=2592000 confirmed via header inspection.
