// New Findings

M1 MEDIUM Host Header Injection in OAuth Discovery Endpoint

File: internal/handler/oauth/handler.go:494–514

The GET /.well-known/oauth-authorization-server endpoint constructs the entire discovery document — issuer, authorization_endpoint, token_endpoint, jwks_uri — by trusting r.Host directly:

// Use the configured issuer from the token issuer when available,
// rather than trusting the Host header which can be spoofed.
scheme := "https"
if r.TLS == nil && (strings.HasPrefix(r.Host, "localhost") || ...) {
    scheme = "http"
}
issuer := scheme + "://" + r.Host  // ← r.Host still used despite comment

The code comment acknowledges the correct fix but it was never implemented. An attacker who can control the Host header receives a discovery document pointing all OAuth endpoints to their domain.

Attack scenario (dev/staging):
An OAuth client using RFC 8414 auto-discovery fetches /.well-known/oauth-authorization-server with a spoofed Host: evil.attacker.com header. The returned document contains "authorization_endpoint": "https://evil.attacker.com/oauth/authorize". The client begins sending authorization requests — including user credentials — to the attacker’s server.
Production mitigating factor: Cloudflare Tunnel normalizes the Host header before forwarding to the origin, making injection from the internet impractical. However, the vulnerability is fully exploitable on local, staging, and CI instances, and also if the origin is ever directly reachable.
Fix: Use the configured issuer string from TokenIssuer.issuer (set at startup from env config) instead of r.Host. Expose it via a method and pass it to the handler at construction time. The comment already says this is the right approach — it just needs to be implemented.
M2 MEDIUM RequireScope / RequireAudience Not Applied to Any Route

Files: internal/auth/middleware.go:97–154, internal/handler/api/router.go

Both RequireScope and RequireAudience middleware are fully implemented but not wired to any route in the identity server. All routes use only RequireAuth and optionally RequireAdmin:

// Every protected route looks like this — no scope or audience check:
mux.Handle("GET /api/v1/users/{id}", auth.RequireAuth(issuer, http.HandlerFunc(uh.get)))

Audience gap

A service token issued to any client with any audience (e.g., aud: "my-api") is accepted by RequireAuth without checking whether its audience matches the identity server. RequireAudience exists precisely to prevent cross-service token replay, but it is never invoked.

In practice, the identity server’s own routes are safe because all handlers either check if claims == nil (blocking service tokens) or are behind RequireAdmin (which also blocks them). However, this creates two concrete risks:

  1. Consuming services: Any downstream service that protects its own endpoints using this identity server’s JWTs must apply RequireAudience themselves to prevent token replay. Nothing in the API, docs, or middleware wiring signals that this is required.
  2. Scope is decorative: Service tokens carry a scope claim that is never enforced. A token with scope: "read:users" has identical access to one with scope: "write:admin" from the identity server’s perspective.
Fix: Apply RequireAudience("id.swee.net") (or the configured issuer hostname) to all identity server API routes. Apply RequireScope to routes where scope differentiation is desired. Add clear documentation that consuming services must apply both middleware to their own protected endpoints.
L1 LOW Service Token Type-Confusion in authorizePasskey (No Auth Bypass)

File: internal/handler/oauth/handler.go:258, internal/auth/jwt.go:236–268

authorizePasskey bridges the WebAuthn ceremony into the OAuth flow by accepting a user access_token and issuing an authorization code. It calls h.tokenIssuer.Parse() directly — not through RequireAuth middleware:

claims, err := h.tokenIssuer.Parse(accessToken)
if err != nil {
    errResp(http.StatusUnauthorized, "invalid_token", "Invalid or expired token.")
    return
}
// ... no IsActive check ...
rawCode, err := h.svc.AuthorizeByUserID(clientID, redirectURI, claims.UserID, ...)

The underlying parseWithKey function has no typ header check — unlike ParseServiceToken which rejects tokens without typ: "at+jwt". A service token (which has sub = clientID, typ: "at+jwt") passes through parseWithKey with TokenClaims{UserID: clientID}.

AuthorizeByUserID does not validate that userID is a real user — it inserts an auth code directly. When the attacker calls ExchangeCode, IssueTokensForUser(clientID) → GetByID(clientID) → ErrNotFound → exchange fails. No auth bypass.

Observable side effects:
1. A garbage auth code is written to the DB with user_id = oauth_client_id.
2. An oauth_authorize_success audit event is emitted with a client ID as the user ID.
Fix: Add a typ check to parseWithKey that rejects service tokens (tokens where typ == "at+jwt"), consistent with ParseServiceToken’s existing check in the other direction.
L2 LOW Empty Audience on client_credentials Client Causes Silent 500

Files: internal/handler/admin/handler.go:683–715, internal/auth/jwt.go:114

The admin UI allows creating a client_credentials OAuth client with an empty audience field — no validation is performed. However, MintServiceToken returns an error if Audience == "":

func (ti *TokenIssuer) MintServiceToken(claims domain.ServiceTokenClaims, ttl time.Duration) (string, error) {
    if claims.Audience == "" {
        return "", errors.New("audience is required for service tokens")
    }

Any subsequent POST /oauth/token with grant_type=client_credentials for that client returns a 500 Internal Server Error. The client has no indication the issue is a configuration problem rather than a server fault.

Fix: In the admin UI create/edit handlers, validate that audience is non-empty when grant_types includes client_credentials. Return a user-facing validation error rather than a silent 500.
L3 LOW verifyAdminPassword Endpoints Under General Rate Limiter (30/min), Not Strict (5/min)

Files: cmd/server/main.go:336–337, internal/handler/admin/router.go:61–74

Only POST /admin/login is wrapped with wrapAuth (5/min strict limiter). All other admin routes fall under /admin/, which receives only the general 30/min rate limiter:

mux.Handle("POST /admin/login", wrapAuth(adminRouter))  // 5/min strict
mux.Handle("/admin/", adminRouter)                       // 30/min general only

These endpoints accept and verify an admin_password field for re-confirmation but are rate-limited at 30/min:

  • POST /admin/users/{id}/edit
  • POST /admin/users/{id}/delete
  • POST /admin/oauth/{id}/edit
  • POST /admin/oauth/{id}/delete

The comment at main.go:323 explicitly lists the credential endpoints requiring strict rate limiting — but doesn’t include the re-confirmation endpoints, suggesting they were overlooked.

Precondition: Requires a valid admin session cookie and CSRF token. An attacker who has obtained both (e.g., via XSS or session hijacking but without knowing the password) can attempt 30 password guesses per minute rather than 5.
Fix: Apply wrapAuth to POST routes that call verifyAdminPassword, or implement a per-session failed-confirmation lockout counter.

// Informational

I1 INFO EC Private Key Stored Plaintext in SQLite

The ES256 private key is stored as PEM in the metadata table, unencrypted — the same design as the prior HS256 secret. A database file compromise grants full token forgery capability. This is an accepted architectural trade-off; noting it for completeness.

I2 INFO JWKS kid Not Used to Select Parse Key

Tokens include a kid header derived from sha256(DER(pubkey))[:8]. However, Parse and ParseServiceToken ignore kid and always try the current key first, then the previous key. Not a security issue, but a missed optimization and deviation from the intent of key IDs in the JWKS spec.
