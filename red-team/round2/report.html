// Detailed Findings

HIGH 1. Refresh Token Rotation Race Condition (TOCTOU) Code
internal/service/auth_service.go:150-192 • internal/store/token_store.go:57-91

The Refresh() function reads the token via GetByHash() and checks tok.IsRevoked outside of any transaction. It then calls issueTokens() → Rotate() which starts its own transaction. Two concurrent requests can both read the token as not-revoked before either begins its write transaction.

Race Timeline: Goroutine A: GetByHash(hash) → tok.IsRevoked == false [releases DB conn] Goroutine B: GetByHash(hash) → tok.IsRevoked == false [releases DB conn] Goroutine A: Rotate(old, newA) → UPDATE is_revoked=1, INSERT newA, COMMIT Goroutine B: Rotate(old, newB) → UPDATE is_revoked=1 (no-op), INSERT newB, COMMIT Result: TWO valid refresh tokens from the same parent Theft detection is defeated — neither token triggers family compromise

Despite SetMaxOpenConns(1) serializing SQL calls, Go goroutines interleave between SQL operations. The Rotate() transaction only protects the revoke+insert pair, not the preceding read+validate.

Compounding factor: The comment on line 56 of token_store.go claims "BEGIN IMMEDIATE transaction" but the code uses plain Begin() which is DEFERRED in SQLite.

// auth_service.go:150-192 — Refresh()
func (s *AuthService) Refresh(rawRefreshToken string) (*LoginResult, error) {
    tokenHash := HashToken(rawRefreshToken)

    tok, err := s.tokens.GetByHash(tokenHash)   // READ — outside any transaction
    // ...
    if tok.IsRevoked {                           // CHECK — stale by race time
        return nil, ErrTokenFamilyCompromised
    }
    // ... validate expiry, user active ...

    return s.issueTokens(user, loginArgs{        // WRITE — Rotate() starts its own tx
        oldTokenID: tok.ID,
        familyID:   tok.FamilyID,
    })
}
// token_store.go:56-58 — Rotate()
// Rotate atomically revokes oldTokenID and inserts newToken within a single
// BEGIN IMMEDIATE transaction to prevent concurrent refresh races.
func (s *TokenStore) Rotate(oldTokenID string, newToken *domain.RefreshToken) error {
    tx, err := s.db.DB().Begin()  // ← plain Begin(), NOT BEGIN IMMEDIATE

Impact: An attacker who intercepts a refresh token can race the legitimate user's refresh request. If both succeed, the attacker has their own valid token chain that won't trigger token_family_compromised when used.

Recommended Fix

Move the GetByHash + IsRevoked check into the Rotate transaction so the entire read-validate-rotate sequence is atomic. Also change Begin() to BEGIN IMMEDIATE to match the documented intent.

MEDIUM 2. No Rate Limiting — Still Present Live
All routers — no rate limiting middleware exists

Confirmed live: 8 rapid sequential POST requests to /api/v1/auth/login all returned HTTP 401 with no 429, no Retry-After, no X-RateLimit-* headers, no delay escalation.

This was finding #2 in Round 1. It remains unfixed.

Recommended Fix

Add per-IP rate limiting middleware, especially on /api/v1/auth/login, /admin/login, /oauth/authorize, and /oauth/token. Return 429 with Retry-After header after threshold.

MEDIUM 3. CF-Connecting-IP Trusted Unconditionally Code
internal/handler/api/auth.go:142-145 • internal/handler/oauth/handler.go:264-267 • internal/handler/admin/handler.go:213-220

All three extractClientIP functions blindly trust the CF-Connecting-IP header:

func extractClientIP(r *http.Request) string {
    if cf := r.Header.Get("CF-Connecting-IP"); cf != "" {
        return cf
    }
    host, _, _ := net.SplitHostPort(r.RemoteAddr)
    return host
}

Impact: If the server is ever reachable directly (bypassing Cloudflare Tunnel), an attacker can forge all audit log IP addresses. Even with Cloudflare in place, the code has no verification that the request actually traversed Cloudflare.

Through the current Cloudflare Tunnel deployment, CF overwrites this header with the real client IP, providing partial protection. But the trust model is fragile.

Recommended Fix

Only trust CF-Connecting-IP when a config flag confirms Cloudflare is in the request path. Validate against Cloudflare's published IP ranges, or rely on RemoteAddr as the default.

MEDIUM 4. Auth Code Error Oracle Code Live
internal/handler/oauth/handler.go:186-193

The /oauth/token endpoint returns distinct error messages that allow an attacker to determine auth code state:

Code Stateerror_description
Never existed"The authorization code is invalid."
Already used"The authorization code has already been used."
Expired"The authorization code has expired."

Impact: Attacker can determine whether a code was ever valid, detect if a race attack succeeded, and infer timing of code issuance.

Recommended Fix

Return a single generic message for all auth code failures: "The authorization code is invalid or has expired."

MEDIUM 5. Admin Password Logged to systemd Journal Code
cmd/server/seed.go:48-56 • deploy/install.sh:97

On first run without explicit credentials, the auto-generated admin password is printed to stdout via log.Printf:

log.Printf("  Password: %s", password)  // persists in journald

The install script tells operators to retrieve it via sudo journalctl -u identity | grep Password. The password persists in the journal with default retention, accessible to any user with journal read access.

Recommended Fix

Write the initial password to a temporary file with 0600 permissions, or require setting it via environment variable. Delete the file after first admin login.

MEDIUM 6. No Re-authentication for Destructive Admin Operations Code
internal/handler/admin/handler.go (multiple handlers)

An admin session (8-hour JWT cookie) grants unrestricted access to all destructive operations without requiring password re-entry:

  • Delete any user (including other admins)
  • Delete any OAuth client
  • Modify user roles and active status
  • Trigger database backups

Impact: A stolen admin session cookie (XSS, network sniffing, physical access) gives an attacker 8 hours of full administrative control.

Recommended Fix

Require password confirmation for: user deletion, role changes, OAuth client deletion, and any operation that could lock out other admins.

MEDIUM 7. OAuth Client Deletion Not Audit-Logged Code
internal/handler/admin/handler.go:560-568

When deleting an OAuth client, no audit event is recorded. Compare to user deletion which calls h.userSvc.Delete(id, h.auditMeta(r)). A compromised admin could delete OAuth clients — breaking downstream applications — without leaving any trace in the audit log.

func (h *adminHandler) oauthDeletePost(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")
    // No audit event recorded
    if err := h.oauthClients.Delete(id); err != nil { ... }
    http.Redirect(w, r, "/admin/oauth", http.StatusSeeOther)
}

Recommended Fix

Add an audit event before the redirect: h.recordEvent(domain.EventOAuthClientDeleted, ...)

INFO 15. Auth Code Exchange TOCTOU (Mitigated)
internal/service/oauth_service.go:104-139 • internal/store/oauth_code_store.go:51-64

Same TOCTOU pattern as the refresh token race: GetByHash + UsedAt == nil check outside transaction, then MarkUsed separately. However, MarkUsed uses WHERE used_at IS NULL and checks RowsAffected, so the second concurrent request correctly fails at the DB level.

This is a defence-in-depth success. The DB-level check catches the race. Documented here because the pattern is fragile — removing the RowsAffected check would reopen the vulnerability.
