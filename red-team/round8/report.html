N1 MEDIUM M1 Fix Regression — Discovery Document Serves Invalid Non-URL Strings CONFIRMED LIVE

File: internal/config/config.go:118–120, internal/handler/oauth/handler.go:494

How the regression was introduced:
The M1 fix correctly replaced r.Host (which produced https://id.swee.net) with h.tokenIssuer.Issuer(). The Issuer() method returns the configured JWT_ISSUER env var, which falls back to SITE_NAME, which is set to "id.swee.net" in production — a bare hostname with no scheme.

The discovery document served live at /.well-known/oauth-authorization-server:

{
  "issuer": "id.swee.net",
  "authorization_endpoint": "id.swee.net/oauth/authorize",
  "token_endpoint": "id.swee.net/oauth/token",
  "jwks_uri": "id.swee.net/.well-known/jwks.json",
  ...
}

RFC 8414 §2 requires the issuer to be a URL using the https scheme, and all endpoint values to be absolute URIs. "id.swee.net/oauth/authorize" is not an absolute URL — it has no scheme. In a browser, new URL("id.swee.net/oauth/authorize") throws TypeError: Invalid URL. OAuth client libraries using auto-discovery (RFC 8414) will either reject the document or silently build broken endpoint URLs.

The JWTs themselves are unaffected — both sides agree on iss: "id.swee.net" — but any integration point that relies on the discovery document for endpoint configuration will fail.

Fix: Set JWT_ISSUER=https://id.swee.net in the production environment. Optionally add a startup validation check that the issuer starts with https://, and update the CLAUDE.md / deployment docs to document this requirement.
N2 LOW CSP Blocks Inline Redirect Script — OAuth Password Flow Redirect Broken

File: internal/ui/templates/oauth_redirect.html, cmd/server/main.go:426

The server sends script-src 'self' (no 'unsafe-inline') on all responses. The OAuth completion page contains an inline script for automatic redirect:

<!-- oauth_redirect.html -->
<a id="redirect-link" href="{{.RedirectURL}}">Click here if you are not redirected</a>
<script>window.location.replace(document.getElementById('redirect-link').href);</script>
                                                                 ↑ blocked by script-src 'self'

This inline script is blocked by the CSP in all modern browsers. When a user completes the password-based OAuth login flow, the "Redirecting to app…" page shows but no automatic redirect occurs — the user must click the fallback link manually.

The passkey flow is unaffected: it uses window.location.href = data.redirect_uri inside the external passkey-login.js file and never hits this page.

Secondary effect: The authorization code and state parameter are visible in the URL of the fallback link while the user is on the page. Auth codes are single-use and expire quickly, so this is a minor concern.
Fix: Move the redirect logic to /static/oauth-redirect.js (same-origin, allowed by script-src 'self'), or remove the script entirely since the Accept: application/json passkey path already handles the JS redirect case.
N3 INFO javascript: Redirect URIs Execute via Client-Side Redirect

Files: internal/ui/static/passkey-login.js:53, internal/ui/templates/oauth_redirect.html:5

Both redirect paths perform client-side navigation without scheme validation:

// passkey-login.js
window.location.href = data.redirect_uri;   // executes javascript: URIs

// oauth_redirect.html
<a href="{{.RedirectURL}}">...</a>          // template.URL() skips sanitizer

If an admin registers an OAuth client with a javascript: scheme redirect URI, any user who goes through the authorization flow would execute the attacker’s JavaScript in their browser context on id.swee.net. The containsURI exact-match check passes (the URI is registered), and the server returns it in the redirect response.

Precondition: Admin access required to register the client. Not an independent attack vector.

Fix: In passkey-login.js, validate the scheme before navigating: if (!/^https?:\/\//i.test(url)) throw new Error('Invalid redirect'). Optionally, reject javascript: and data: redirect URIs at registration time in the admin UI.
N4 MEDIUM Introspect Endpoint Discloses Any Token to Any Authenticated Client

File: internal/handler/oauth/handler.go:426–491

RFC 7662 §2.1 states the authorization server "SHOULD determine whether the token is valid and whether it belongs to the requesting client." The /oauth/introspect endpoint authenticates the requesting client correctly but performs no ownership check before returning full token claims.

Any authenticated OAuth client can present a token belonging to a different client and receive full claims:

// line 456 — no check that sc.ClientID == creds.ClientID
if sc, err := h.tokenIssuer.ParseServiceToken(token); err == nil {
    resp := map[string]any{
        "active":    true,
        "sub":       sc.ClientID,    // reveals owning client
        "client_id": sc.ClientID,
        "scope":     sc.Scope,       // reveals granted scopes
        "jti":       sc.JTI,
        "exp":       sc.ExpiresAt,
        "iat":       sc.IssuedAt,
    }
    jsonOK(w, resp)   // returned to any authenticated client
    return
}
// line 478 — same issue for user tokens
if uc, err := h.tokenIssuer.Parse(token); err == nil {
    jsonOK(w, map[string]any{
        "active":   true,
        "sub":      uc.UserID,
        "username": uc.Username,    // reveals username to any client
    })
    return
}

A compromised or malicious client that legitimately receives any token (e.g. via token passing in an API call) can introspect it to learn: the owning service client's scopes, or the user's UUID and username. This violates tenant isolation on a shared authorization server.

Fix: Add ownership validation before returning claims:
• Service tokens: if sc.ClientID != creds.ClientID { jsonOK(w, map[string]any{"active": false}); return }
• User tokens: check that the token's aud matches the requesting client's registered audience.
N5 LOW Introspect Response Missing aud Claim for Service Tokens

File: internal/handler/oauth/handler.go:457–472

RFC 7662 §2.2 says the introspect response SHOULD include aud if present in the token. Service tokens (client credentials flow) are issued with a configured audience. The introspect response omits it:

resp := map[string]any{
    "active":     true,
    "sub":        sc.ClientID,
    "client_id":  sc.ClientID,
    "scope":      sc.Scope,
    "token_type": "Bearer",
    "jti":        sc.JTI,
    // "aud" missing — present in the JWT but not reflected here
}

Resource servers that rely on introspection to enforce audience cannot do so. They either trust the token blindly or must re-parse the JWT themselves, defeating the purpose of token introspection.

Fix: Include "aud": sc.Audience in the introspect response for service tokens (requires ServiceClaims to expose the Audience field, or retrieve it from the registered client record).
N6 INFO OAuth Client ID Accepts Arbitrary Characters

File: internal/handler/admin/handler.go:677

The admin oauthNewPost handler validates only that the client ID is non-empty; no character set or format restriction is applied:

id := strings.TrimSpace(r.FormValue("id"))
if id == "" || name == "" {
    // error
}
// id can contain spaces, slashes, null bytes, etc.

Malformed client IDs (e.g. containing /, \x00, or Unicode control characters) could cause unexpected behavior in URL construction, logging pipelines, or future code paths that embed the client ID in structured output. Admin-controlled, not independently exploitable.

Fix: Validate client IDs at registration time against a safe character set: ^[a-zA-Z0-9._-]+$ (or similar).
