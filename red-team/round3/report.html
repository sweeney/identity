// Detailed Findings

HIGH 1. Rate Limiter Bypass via Dual ExtractClientIP Code
internal/ratelimit/ratelimit.go:110-121 • internal/httputil/clientip.go:11-22

The rate limiter package has its own ExtractClientIP() that always trusts CF-Connecting-IP, while the shared httputil.ExtractClientIP() correctly respects the TRUST_PROXY config:

// ratelimit.go:112 — ALWAYS trusts (VULNERABLE)
func ExtractClientIP(r *http.Request) string {
    if cf := r.Header.Get("CF-Connecting-IP"); cf != "" {
        return cf  // ← unconditional trust, ignores TRUST_PROXY
    }
}

// httputil/clientip.go:11 — respects config (CORRECT)
func ExtractClientIP(r *http.Request, trustProxy string) string {
    if trustProxy == "cloudflare" {  // ← checks config first
        if cf := r.Header.Get("CF-Connecting-IP"); cf != "" {
            return cf
        }
    }
}

Impact: An attacker rotates the CF-Connecting-IP header per request, getting a fresh rate limit bucket each time. This completely bypasses all rate limiting — the very fix from Round 2 finding #2.

Attack: Attacker → CF-Connecting-IP: 192.0.2.1 → POST /api/v1/auth/login → 401 (bucket 1: 1/5) Attacker → CF-Connecting-IP: 192.0.2.2 → POST /api/v1/auth/login → 401 (bucket 2: 1/5) Attacker → CF-Connecting-IP: 192.0.2.3 → POST /api/v1/auth/login → 401 (bucket 3: 1/5) ... Attacker → CF-Connecting-IP: 192.0.2.N → POST /api/v1/auth/login → 401 (bucket N: 1/5) Result: Unlimited password attempts, each IP never exceeds its 5-request limit

Recommended Fix

Delete ratelimit.ExtractClientIP(). Have Limiter accept a trustProxy parameter and call httputil.ExtractClientIP(r, trustProxy). One function, one behavior.

MEDIUM 2. POST /oauth/authorize Missing Strict Rate Limit Code
cmd/server/main.go:287-290

The strict auth rate limiter (5 req/min) wraps POST /api/v1/auth/login, POST /oauth/token, and POST /admin/login. But POST /oauth/authorize — which also accepts username+password — only gets the general rate limiter (30 req/min).

mux.Handle("POST /api/v1/auth/login", wrapAuth(apiRouter))     // strict: 5/min
mux.Handle("/api/v1/", apiRouter)
mux.Handle("POST /oauth/token", wrapAuth(oauthRouter))         // strict: 5/min
mux.Handle("/oauth/", oauthRouter)                            // general only: 30/min
mux.Handle("POST /admin/login", wrapAuth(adminRouter))         // strict: 5/min
mux.Handle("/admin/", adminRouter)

An attacker can brute-force credentials at 6x the rate of the login endpoint by targeting /oauth/authorize instead. Combined with finding #1 (IP spoofing), this becomes unlimited.

Recommended Fix

Add mux.Handle("POST /oauth/authorize", wrapAuth(oauthRouter)) before the catch-all /oauth/ route.

MEDIUM 3. OAuth Client Edit Not Protected by Re-auth Code
internal/handler/admin/handler.go

The re-auth feature covers user deletion, role changes, deactivation, and OAuth client deletion — but not OAuth client editing. An attacker with a hijacked admin session (2-hour window) can silently modify an OAuth client's redirect_uri to an attacker-controlled domain.

This enables silent OAuth token theft: all users who subsequently authenticate via that client will have their authorization codes redirected to the attacker. This is arguably more dangerous than deletion, which causes a visible outage.

Recommended Fix

Require password re-auth for OAuth client edits, especially redirect_uri changes.
