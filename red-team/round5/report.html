MEDIUM-HIGH R5-01: Username Enumeration via WebAuthn Login/Begin

The POST /api/v1/webauthn/login/begin endpoint produces three distinguishable responses that reveal user state: (1) nonexistent user returns a fake discoverable challenge, (2) existing user with no passkeys returns webauthn_no_credentials error, (3) existing user with passkeys returns a real challenge containing allowCredentials with credential IDs.

Attack Scenario

1. Attacker sends POST /api/v1/webauthn/login/begin with a target username.
2. If the response is HTTP 400 with webauthn_no_credentials, the user exists but has no passkeys.
3. If the response is HTTP 200 with allowCredentials: [] (empty), the user does not exist (fake discoverable challenge).
4. If the response is HTTP 200 with populated allowCredentials, the user exists and has passkeys. The credential IDs are leaked, which can be used for targeted attacks.
5. This oracle is not mitigated by the dummy-bcrypt timing fix applied to password login.

Evidence

Source: internal/service/webauthn_service.go lines 222–228 (fake challenge for nonexistent user), lines 238–239 (error on exists-but-no-passkeys).
Live test: poc-username-enum.py — confirmed 3-way oracle working against production. Rate limit is 5/min but supports slow enumeration over time.

Recommended Fix

Unify the response for all three cases: always return a discoverable-style challenge (HTTP 200), regardless of user existence or passkey status. The FinishLogin callback will reject the assertion for nonexistent users. This matches the dummy-bcrypt pattern used for password login.

MEDIUM R5-02: TOCTOU Race on Challenge Consumption

Both FinishRegistration and FinishLogin use defer s.challenges.Delete(challengeID) to clean up challenges. The challenge is fetched, validated, and the cryptographic ceremony is completed before the deferred delete executes. Two concurrent requests with the same challenge_id can both read the challenge before either deletes it.

Attack Scenario

1. User initiates a WebAuthn login, receiving a challenge_id.
2. Attacker intercepts or shares the challenge_id (e.g., via XSS or MitM).
3. Attacker and legitimate user both send FinishLogin simultaneously.
4. Both requests call GetByID — both succeed because the deferred delete has not fired.
5. Both proceed through cryptographic verification. If both have valid assertions, both receive tokens.
6. SQLite single-connection serialization narrows but does not fully close this window, since the Go HTTP server handles requests concurrently and the CPU-bound crypto step releases the DB connection.

Evidence

Source: internal/service/webauthn_service.go line 135 (defer in FinishRegistration), line 290 (defer in FinishLogin).
PoC: poc-challenge-race.py demonstrates the race window.

Recommended Fix

Delete the challenge atomically before performing cryptographic verification. Use a transaction: SELECT + DELETE + commit in one step, then proceed with validation using the fetched data. This ensures the challenge is consumed exactly once.

MEDIUM R5-03: Hardcoded UserPresent/UserVerified Flags

The DomainCredentialToWebAuthn function hardcodes UserPresent: true and UserVerified: true for every credential. The actual authenticator flags from registration are never persisted — the domain model and database schema lack columns for UP and UV. This permanently destroys the security assurance metadata for every credential.

Attack Scenario

1. User registers a basic security key that supports UP but not UV (no biometric).
2. The authenticator correctly reports UV=false during registration.
3. The server discards this flag and stores the credential as if UV=true.
4. If step-up authentication or risk-based access control is later implemented based on UV status, this credential is indistinguishable from a biometric-verified one.
5. A stolen non-biometric key gains the same trust level as a biometric-protected credential.
6. Additionally, if the RP tightens policy from "preferred" to "required", it cannot identify which existing credentials actually had UV.

Evidence

Source: internal/auth/webauthn.go lines 34–39 (hardcoded flags).
internal/domain/webauthn.go lines 8–22 (missing fields).
internal/service/webauthn_service.go lines 178–192 (flags not extracted at registration).
internal/store/webauthn_credential_store.go lines 24–54 (not stored in DB).

Recommended Fix

1. Add user_present and user_verified columns to the webauthn_credentials table.
2. Add corresponding fields to the domain model.
3. Persist the actual flags from credential.Flags at registration time.
4. Restore the actual flags in DomainCredentialToWebAuthn instead of hardcoding.
5. Migrate existing credentials conservatively: user_present=true, user_verified=false.

MEDIUM R5-04: No Credential Registration Limit

There is no cap on the number of passkeys a user can register. An attacker with a stolen JWT can register unlimited rogue passkeys as persistent backdoors. At the general rate limit of 30 req/min (registration is not behind the strict auth limiter), an attacker can register ~15 passkeys/minute, yielding ~225 passkeys within a single 15-minute access token lifetime.

Attack Scenario

1. Attacker obtains a valid JWT for a target user (via XSS, token theft, or compromised session).
2. Attacker scripts rapid register/begin + register/finish calls, each with a different authenticator.
3. ~225 rogue passkeys are registered within the JWT lifetime.
4. Victim rotates password and revokes refresh tokens — rogue passkeys survive.
5. Attacker retains persistent login access via any of the 225 passkeys.
6. Victim must manually discover and delete every rogue credential to fully recover.

Evidence

Source: internal/service/webauthn_service.go line 72 (BeginRegistration does not check credential count).
internal/db/migrations/003_webauthn.sql — no CHECK constraint or per-user limit.
cmd/server/main.go lines 289–319 — registration endpoints only behind general rate limiter (30/min), not strict (5/min).

Recommended Fix

1. Cap credentials per user (e.g., 25). Check len(existingCreds) in BeginRegistration.
2. Apply the strict auth rate limiter to registration endpoints.
3. Require re-authentication (password or existing passkey) before registering a new passkey, instead of accepting any valid JWT.

LOW R5-05: CORS Origins Auto-Merged into WebAuthn Origins

Lines 173–187 of internal/config/config.go automatically merge all CORS_ORIGINS into WebAuthnRPOrigins. This couples two different trust decisions: "this origin may read API responses" (CORS) and "this origin may use passkeys bound to our RP ID" (WebAuthn authentication). Adding a sibling subdomain to CORS silently grants it WebAuthn ceremony rights.

Attack Scenario

1. Developer adds https://app.swee.net to CORS_ORIGINS for a new SPA.
2. Auto-merge silently adds it to WebAuthnRPOrigins.
3. XSS or compromise on app.swee.net allows attacker to initiate WebAuthn ceremonies using passkeys bound to RP ID swee.net.
4. The browser permits this because the RP ID matches. The server permits it because the origin is in the allowed list.

Evidence

Source: internal/config/config.go lines 173–187 (merge logic with explicit comment).
Current risk: Low (CORS_ORIGINS appears to be empty or id.swee.net only). Risk increases if sibling subdomains are added.

Recommended Fix

Remove the auto-merge logic. Configure WEBAUTHN_RP_ORIGINS and CORS_ORIGINS independently. Add a startup warning if they diverge, to surface the trust boundary difference.

LOW R5-06: Query Parameters on POST Endpoints

The WebAuthn finish endpoints accept challenge_id and name as URL query parameters on POST requests rather than in the request body. Query parameters on POST requests are an anti-pattern: they appear in Referer headers, may be logged by reverse proxies (Cloudflare), and are visible in browser developer tools.

Evidence

Source: internal/handler/api/webauthn.go lines 45–46 (r.URL.Query().Get("challenge_id"), r.URL.Query().Get("name")).
internal/handler/admin/handler.go lines 407–408 (same pattern).
The name parameter could contain PII if users name passkeys after devices.

Recommended Fix

Move challenge_id and name to the POST request body. These are POST endpoints already receiving a JSON body (the WebAuthn attestation/assertion), so wrapping in an envelope or adding fields is trivial.

LOW R5-07: Passkey Name Unsanitized

The passkey name parameter has no length limit, no character filtering, and no input validation. An attacker with a valid session can register a passkey named <script>alert(1)</script> or a multi-megabyte string. XSS is blocked by Go's html/template auto-escaping, but the lack of validation allows UI disruption (layout breakage from very long names, RTL override characters, zero-width abuse).

Evidence

Source: internal/handler/api/webauthn.go line 46 (no validation).
internal/ui/templates/passkeys.html line 14 and user_form.html line 58 ({{.Name}} — auto-escaped, XSS blocked).
No template.HTML() or unsafe functions found in the template FuncMap.

Recommended Fix

Add input validation: limit passkey names to 100 characters, allow only alphanumeric + spaces + basic punctuation, strip control characters.

LOW R5-08: Admin Passkey Login Missing CSRF Token

POST /admin/login/passkey accepts an access_token via form body and sets an admin session cookie, but has no CSRF token validation. Unlike all other admin POST routes, this one is not wrapped in requireCSRF(). The same pattern exists on POST /oauth/authorize/passkey.

Mitigation: The admin_session cookie uses SameSite=Strict, which blocks cross-site form submissions in all modern browsers. The attack also requires a pre-obtained valid JWT access token (15-min TTL, HS256-signed). The combination makes exploitation unrealistic.

Evidence

Source: internal/handler/admin/router.go line 48 (no requireCSRF).
internal/handler/admin/handler.go lines 258–296 (loginPasskey handler).
internal/handler/admin/handler.go lines 134–143 (SameSite=Strict on session cookie).

Recommended Fix

Add an Origin/Referer header check as defense-in-depth. A full CSRF token is difficult here since no session exists yet, but verifying the request origin matches https://id.swee.net closes the gap.

INFO R5-09: HSTS 30-Day Max-Age, No Preload

Live server returns max-age=2592000 (30 days). Code shows max-age=63072000 (2 years) — likely Cloudflare override or deployment lag. Missing includeSubDomains and preload directives. Carried forward from Round 4.

Live test: curl -sI https://id.swee.net/ | grep -i strict confirms 30-day max-age.
INFO R5-10: RP ID Scoped to swee.net (By Design)

The WebAuthn RP ID is swee.net rather than id.swee.net. This means passkeys are cryptographically bound to the parent domain and can be exercised by any origin under *.swee.net, subject to the RPOrigins server-side check. This broadens the trust boundary permanently — changing the RP ID would invalidate all existing passkeys.

Current mitigation: The RPOrigins allowlist restricts which origins can complete ceremonies. As long as only https://id.swee.net is listed, sibling subdomains are rejected.

Source: internal/auth/webauthn.go — RPID: "swee.net".
internal/config/config.go lines 148–187 (RP ID and origins configuration).

  • No XSS — Go template auto-escaping, strong CSP, no unsafe template functions

// Conclusion

The core identity server remains well-hardened after four rounds of testing. The WebAuthn/passkey implementation is functional and the cryptographic ceremony handling is correct, but the surrounding infrastructure has gaps in credential lifecycle management (R5-03, R5-04), information leakage (R5-01), and concurrency safety (R5-02). These are typical first-implementation issues and are straightforward to remediate.
